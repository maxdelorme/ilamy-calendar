# Development Log - 2026-05-19

## Changes

- **[callbacks / recurrence]**: Scoped recurring deletes (`this`, `this and following`) now notify persistence via `onEventUpdate` with the updated base series row (EXDATE / UNTIL), not `onEventDelete`. `onEventDelete` is used only for `all`, and fires once with the stored base series row (never per override, never synthetic instance ids such as `<id>_0`).
- **[recurrence-handler]**: `deleteRecurringEvent` now returns `{ events, updatedRecurringEvent?, deletedEvents? }` so the engine can emit the correct update/delete payload without re-scanning.

## Notes

Consumers that treated every recurring delete as a row removal should switch: EXDATE / UNTIL edits are updates to the series master; only `all` is a delete. Scope `all` notifies one `onEventDelete` on the base row (see `2026-05-20.md` for override handling and final uid contract).
