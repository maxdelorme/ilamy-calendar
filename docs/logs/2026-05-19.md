# Development Log - 2026-05-19

## Changes

- **[recurrence / scope all]**: Scope `"all"` now always collapses stored data for that series to the updated base event. Detached overrides for the series UID are removed, all EXDATEs are cleared, and unrelated events/series remain untouched.
- **[recurrence / all-time anchoring]**: Scope `"all"` time updates now always apply the submitted wall-clock time to the base event's anchor day instead of branching between generated and detached targets.
- **[recurrence / duration clarity]**: `applyAllScopeUpdates` now computes the new end with an explicit `updates.end` branch, preserving base duration when no end is submitted.
- **[tests]**: Regressions cover collapse from detached targets, generated instances with date-plus-time moves, and generated instances with field-only updates.
- **[log sync]**: Refreshed this log after the latest source/test edits so it remains newer than the modified recurrence files.
- **[recurrence / scope this + detached]**: Re-applying scope `"this"` when the drag target is already a stored override now keeps the original occurrence key (`recurrenceId`), removes the previous override row, and avoids duplicate EXDATEs so the event does not appear twice after a second drag.
- **[recurrence / readability]**: Scope `"this"` recurrence ID selection now uses an explicit `updatingOverrideEvent` branch: existing overrides keep `targetEvent.recurrenceId`, generated instances use `targetEvent.start.toISOString()`.

## Notes

The old branch that patched detached override rows for scope `"all"` was removed. The invariant is now simpler: after an edit/update with scope `"all"`, a recurring series has one persisted base row with no exceptions plus any unrelated events from other series. `applyAllScopeUpdates` also uses one start-time rule for every target type: keep the base anchor date and copy the submitted wall-clock time. When no end is submitted, it preserves the base event duration.

For scope `"this"` on an existing detached override, `instanceAnchor` represents the original series occurrence key, not the override's current displayed start. Keeping `recurrenceId` tied to that anchor lets repeated drag-and-drop edits replace the same stored override instead of creating duplicate detached rows. The code now spells out that rule with `updatingOverrideEvent`: override targets keep `targetEvent.recurrenceId`, while generated targets derive the key from `targetEvent.start`.
